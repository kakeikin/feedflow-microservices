# FeedFlow Microservices — Phase 1: Service Foundation Design

## Overview

FeedFlow Microservices is the second-generation version of FeedFlow. The original FeedFlow was a monolithic personalized feed backend (FastAPI + PostgreSQL + Redis + RQ). This version redesigns the system into independently deployable services to improve fault isolation, maintainability, and scalability.

Phase 1 focuses on service decomposition, synchronous HTTP-based inter-service communication, Docker Compose orchestration, and integration testing. It deliberately excludes RabbitMQ, Redis, Kafka, Prometheus, and authentication — those are Phase 2+.

**New repository:** `feedflow-microservices/` (sibling to original `FeedFlow/`)

---

## Architecture

```
Client → Feed Service (8005)
           │
           ├─ GET /videos/trending  → Video Service (8002)
           └─ POST /rank            → Ranking Service (8004)
                                         │
                                         ├─ GET /users/{id}/interests → User Service (8001)
                                         └─ GET /videos/{id}          → Video Service (8002)

Client → Event Service (8003) → event-db (5435)
Client → User Service (8001)  → user-db  (5433)
Client → Video Service (8002) → video-db (5434)
```

All inter-service communication is synchronous HTTP using `httpx.AsyncClient` with `timeout=3.0`.

Ranking Service and Feed Service are stateless — they hold no database and call other services for data.

## Services

### 1. User Service — port 8001

**Responsibility:** Manage users and user interest profiles.

**Database:** `user-db` (PostgreSQL, port 5433)

**Tables:**

```sql
users (
  id           UUID PRIMARY KEY,   -- generated by Python uuid.uuid4()
  email        TEXT UNIQUE NOT NULL,
  display_name TEXT NOT NULL,
  created_at   TIMESTAMP NOT NULL DEFAULT now()
)

user_interests (
  id         UUID PRIMARY KEY,   -- generated by Python uuid.uuid4()
  user_id    UUID NOT NULL REFERENCES users(id),
  tag        TEXT NOT NULL,
  score      FLOAT NOT NULL DEFAULT 1.0,
  updated_at TIMESTAMP NOT NULL DEFAULT now(),
  UNIQUE (user_id, tag)
)
```

**UUID generation:** All primary keys are generated in application code using `str(uuid.uuid4())`. PostgreSQL `gen_random_uuid()` is not used (avoids needing the `pgcrypto` extension).

**API:**

| Method | Path | Description |
|---|---|---|
| GET | /health | Health check |
| POST | /users | Create user |
| GET | /users/{user_id} | Get user |
| GET | /users/{user_id}/interests | List interest tags |
| POST | /users/{user_id}/interests | Add or update an interest tag |

`POST /users` request:
```json
{ "email": "jiaxin@example.com", "display_name": "Jiaxin" }
```

`POST /users/{user_id}/interests` request:
```json
{ "tag": "ai", "score": 0.9 }
```
Response (201):
```json
{ "tag": "ai", "score": 0.9 }
```
Upsert semantics: if the `(user_id, tag)` pair already exists, update `score` and `updated_at`. This allows seeding interests for new users.

`GET /users/{user_id}/interests` response:
```json
[{ "tag": "ai", "score": 0.9 }, { "tag": "finance", "score": 0.7 }]
```

### 2. Video Service — port 8002

**Responsibility:** Manage video metadata and video statistics.

**Database:** `video-db` (PostgreSQL, port 5434)

**Tables:**

```sql
videos (
  id               UUID PRIMARY KEY,   -- generated by Python uuid.uuid4()
  title            TEXT NOT NULL,
  creator_id       UUID NOT NULL,
  tags             TEXT[] NOT NULL,
  duration_seconds INTEGER NOT NULL,
  created_at       TIMESTAMP NOT NULL DEFAULT now()
)

video_stats (
  id              UUID PRIMARY KEY,   -- generated by Python uuid.uuid4()
  video_id        UUID NOT NULL UNIQUE REFERENCES videos(id),
  views           INTEGER NOT NULL DEFAULT 0,
  likes           INTEGER NOT NULL DEFAULT 0,
  skips           INTEGER NOT NULL DEFAULT 0,
  completion_rate FLOAT NOT NULL DEFAULT 0.0,
  updated_at      TIMESTAMP NOT NULL DEFAULT now()
)
```

**UUID generation:** All primary keys are generated in application code using `str(uuid.uuid4())`.

**API:**

| Method | Path | Description |
|---|---|---|
| GET | /health | Health check |
| POST | /videos | Create video (atomically creates video_stats) |
| GET | /videos | List all videos |
| GET | /videos/trending | Top videos by engagement |
| GET | /videos/{video_id} | Get video + stats |

**`POST /videos` atomic creation:** Inserting a video also inserts a `video_stats` row (with all defaults) in the same database transaction. This guarantees every video has stats, so `GET /videos/trending` (which joins `video_stats`) never misses newly created videos.

**Route ordering note:** In `routes.py`, the `/videos/trending` route must be registered before `/videos/{video_id}`. FastAPI matches routes in definition order — if `{video_id}` is defined first, a request to `/videos/trending` will be treated as a lookup for video ID `"trending"`.

`GET /videos/trending` returns up to 20 videos ordered by:
```sql
ORDER BY video_stats.likes DESC, video_stats.views DESC, videos.created_at DESC
LIMIT 20
```

`GET /videos/{video_id}` response includes both video metadata and its stats:
```json
{
  "id": "uuid",
  "title": "Intro to AI Agents",
  "creator_id": "uuid",
  "tags": ["ai", "agents", "backend"],
  "duration_seconds": 120,
  "created_at": "timestamp",
  "stats": {
    "views": 500,
    "likes": 120,
    "skips": 30,
    "completion_rate": 0.72
  }
}
```

### 3. Event Service — port 8003

**Responsibility:** Receive and persist user interaction events with idempotency.

**Database:** `event-db` (PostgreSQL, port 5435)

**Table:**

```sql
events (
  id                  UUID PRIMARY KEY,   -- generated by Python uuid.uuid4()
  user_id             UUID NOT NULL,
  video_id            UUID NOT NULL,
  event_type          TEXT NOT NULL,
  completion_rate     FLOAT,
  watch_time_seconds  INTEGER,
  idempotency_key     TEXT UNIQUE NOT NULL,
  created_at          TIMESTAMP NOT NULL DEFAULT now()
)
```

Allowed `event_type` values: `watch`, `like`, `skip`, `complete`, `share`, `comment`.

**API:**

| Method | Path | Description |
|---|---|---|
| GET | /health | Health check |
| POST | /events | Ingest event |
| GET | /events | List events (filter by ?user_id=) |

`POST /events` request:
```json
{
  "user_id": "uuid",
  "video_id": "uuid",
  "event_type": "like",
  "completion_rate": 0.85,
  "watch_time_seconds": 42,
  "idempotency_key": "user-video-like-001"
}
```

Normal response (201):
```json
{ "id": "uuid", "status": "created" }
```

Duplicate key response (200):
```json
{ "status": "duplicate_ignored" }
```

**Idempotency implementation:** Attempt insert; catch `IntegrityError` on `idempotency_key` unique constraint and return `duplicate_ignored` instead of raising.

### 4. Ranking Service — port 8004

**Responsibility:** Rank a set of candidate videos for a given user using rule-based scoring.

**Stateless** — no database. Calls User Service and Video Service via HTTP.

**API:**

| Method | Path | Description |
|---|---|---|
| GET | /health | Health check |
| POST | /rank | Rank candidate videos |

`POST /rank` request:
```json
{
  "user_id": "uuid",
  "candidate_video_ids": ["uuid1", "uuid2", "uuid3"]
}
```

`POST /rank` response:
```json
[
  {
    "video_id": "uuid1",
    "score": 0.92,
    "reason": "matched user interest tags: ai, backend"
  }
]
```

Results are sorted by `score` descending.

**Scoring formula:**

```
final_score = 0.60 × interest_match_score
            + 0.25 × freshness_score
            + 0.15 × popularity_score
```

All component scores and `final_score` are in **[0.0, 1.0]**.

**Score definitions:**

`interest_match_score` — fraction of a video's tags that appear in the user's interest list, weighted by interest score:
```
interest_match_score = sum(user_interest_score[tag] for tag in video.tags if tag in user_interests)
                       / len(video.tags)
```
If user has no interests or video has no tags: `0.0`.

`freshness_score` — linear decay based on video age:
```
age_hours = (now - video.created_at).total_seconds() / 3600
freshness_score = max(0.0, 1.0 - age_hours / 168)   # 168h = 7 days
```
Videos under 1 hour old score ~1.0. Videos 7+ days old score 0.0.

`popularity_score` — normalized across the candidate set:
```
raw = video_stats.likes
popularity_score = raw / max_likes_in_candidate_set   (0.0 if all have 0 likes)
```

**HTTP calls made by Ranking Service:**
- `GET {USER_SERVICE_URL}/users/{user_id}/interests` — timeout 3.0s
- `GET {VIDEO_SERVICE_URL}/videos/{video_id}` per candidate — timeout 3.0s

**Partial failure handling:** Each candidate video is fetched independently. If a single video fetch fails (timeout, connection error, non-2xx response), that video is skipped and scoring continues with the remaining candidates. A failure on one video never aborts the entire `/rank` call. If all candidates fail, return an empty list.

### 5. Feed Service — port 8005

**Responsibility:** Public feed API. Fetches trending videos and delegates ranking to Ranking Service. Implements graceful degradation.

**Stateless** — no database.

**API:**

| Method | Path | Description |
|---|---|---|
| GET | /health | Health check |
| GET | /feed/{user_id} | Get personalized feed |

`GET /feed/{user_id}` response:
```json
{
  "user_id": "uuid",
  "source": "personalized_ranking",
  "items": [
    {
      "video_id": "uuid",
      "score": 0.91,
      "reason": "matched user interest tags: ai"
    }
  ]
}
```

**Feed generation flow:**
1. Call `GET {VIDEO_SERVICE_URL}/videos/trending` — timeout 3.0s
2. Extract candidate video IDs
3. Call `POST {RANKING_SERVICE_URL}/rank` with `user_id` and candidate IDs — timeout 3.0s
4. Return ranked list with `source: "personalized_ranking"`

**Fallback — if Ranking Service fails (timeout, connection error, non-2xx):**
- Return trending videos directly with `score=0.0` and `reason="trending_fallback"`
- Set `source: "trending_fallback"` in response

This fallback ensures the feed endpoint stays available even when the ranking service is degraded.

## Service-to-Service Communication

All HTTP calls use `httpx.AsyncClient` with `timeout=3.0`.

Service URLs are injected via environment variables:

| Variable | Value in Docker Compose |
|---|---|
| `USER_SERVICE_URL` | `http://user-service:8001` |
| `VIDEO_SERVICE_URL` | `http://video-service:8002` |
| `RANKING_SERVICE_URL` | `http://ranking-service:8004` |

Services never hardcode `localhost`. URL env vars are required at startup.

## Per-Service File Structure

Database-backed services (user, video, event):
```
services/<name>/
  app/
    main.py       # FastAPI app + lifespan (create_all on startup)
    models.py     # SQLAlchemy ORM models
    schemas.py    # Pydantic request/response schemas
    database.py   # async engine + AsyncSession factory
    routes.py     # route handlers
  tests/
    test_routes.py
  Dockerfile
  requirements.txt
```

Stateless services (ranking, feed):
```
services/<name>/
  app/
    main.py       # FastAPI app
    schemas.py    # Pydantic schemas
    clients.py    # httpx calls to upstream services
    routes.py     # route handlers
    ranking.py    # scoring logic (ranking-service only)
  tests/
    test_ranking.py / test_routes.py
  Dockerfile
  requirements.txt
```

## Database Initialization

Each database-backed service initializes its tables on startup using SQLAlchemy's `create_all()`:

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    yield
```

No Alembic in Phase 1. Tables are created if they don't exist; existing tables are untouched.

## Docker Compose

Single file: `infra/docker-compose.yml`

Starts: `user-service`, `video-service`, `event-service`, `ranking-service`, `feed-service`, `user-db`, `video-db`, `event-db`.

Each DB has:
- `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` env vars
- named volume for data persistence
- healthcheck via `pg_isready`

Each service has:
- `build` pointing to its service directory
- `ports` mapping
- `environment` for `DATABASE_URL` (database services) and upstream `*_SERVICE_URL` (ranking, feed)
- `depends_on` with `condition: service_healthy` for its DB

Run command:
```bash
docker compose -f infra/docker-compose.yml up --build
```

## Testing

### Unit Tests (per service)

| Service | What to test |
|---|---|
| user-service | Schema validation, user creation response shape |
| video-service | Schema validation, trending sort logic |
| event-service | Idempotency logic, event_type validation |
| ranking-service | Scoring formula for each component score, final score clamping, fallback when user has no interests |
| feed-service | Fallback path (mock Ranking Service failure returns trending with score=0.0) |

### Integration Tests (root level — assume Docker Compose running)

`tests/integration/test_service_health.py` — GET /health for all 5 services returns 200

`tests/integration/test_feed_flow.py`:
- Test 1: Create user → create video → GET /videos/trending returns the video
- Test 2: GET /feed/{user_id} returns ranked list with `items` array
- Test 3: POST same event twice → second response is `{"status": "duplicate_ignored"}`

Integration tests use `httpx` pointed at `localhost` ports. No pytest fixtures spin up Docker — tests assume the stack is running.

### Feed Service Unit Test — Fallback Path

In `services/feed-service/tests/test_routes.py`, mock the httpx client to raise `httpx.ConnectError` when calling Ranking Service. Assert that the response has `source: "trending_fallback"` and all items have `score: 0.0`. This cannot be tested against the live stack (ranking-service is always up there) so it belongs in unit tests.

## Definition of Done

Phase 1 is complete when:

1. Five FastAPI services exist, each in `services/<name>/`
2. `docker compose -f infra/docker-compose.yml up --build` starts all 8 containers
3. All 5 services respond 200 on `/health`
4. User Service: create and fetch users, add/list interests (upsert)
5. Video Service: create and list videos, trending endpoint returns results ordered by likes/views
6. Event Service: create events, duplicate key returns `duplicate_ignored`
7. Ranking Service: accepts candidate IDs, returns scored list using the defined formula
8. Feed Service: returns personalized feed; falls back to trending if Ranking Service fails
9. Root-level integration tests pass against running Docker Compose stack
10. README explains architecture, service map, and local run commands
